**Visão Geral do Projeto e Regras de Negócio:**
*   **Objetivo:** Sistema para gerenciar a vida dos conviventes (data de entrada, histórico, alocação em quartos, controle de presença).
*   **Modelo de Negócios:** SaaS (Software as a Service) com Multi-Tenancy. Cobrança de mensalidade. O bloqueio de inadimplentes é feito via backend (flag `is_active` na tabela Instituição e bloqueio de token).
*   **Regra Financeira Estrita:** O motor financeiro (Módulo 5) deve considerar como "projeção" estritamente itens com datas futuras em relação ao dia de hoje. O sistema DEVE ignorar parcelas projetadas durante conciliação de dívidas para não inflar totais em aberto.

**Stack de Tecnologia Definitiva:**
*   **Frontend:** React como PWA (Progressive Web App). Estilização via Tailwind CSS. Hospedagem inicial: Vercel.
*   **Backend:** Python com **FastAPI** (Assíncrono, validação Pydantic). Hospedagem inicial: Render.com.
*   **Banco de Dados:** PostgreSQL hospedado no Supabase. Acesso via SQLAlchemy + asyncpg.

**Diretrizes Arquiteturais Rigorosas (Foco em Hardware Precário e Internet Instável):**
*   **Offline-First e Resiliência:** Uso de React Query/SWR e `localStorage`. Ações como chamadas de presença devem funcionar offline e sincronizar em background.
*   **Optimistic UI:** A interface deve refletir ações de salvamento instantaneamente, mascarando a latência da rede.
*   **Performance e Segurança via RBAC:** O backend não envia dados completos para o front filtrar. O FastAPI corta o payload (JSON) com base no Token JWT do usuário. Payloads devem ser minúsculos e usar paginação estrita.

Módulo 1: Core SaaS (Autenticação e Controle)
Este é o coração do seu modelo de negócios. Ele não lida com os conviventes, mas sim com os seus clientes (as ONGs).

O que faz: Gerencia o login, o ID de cada instituição (Tenant) e o isolamento dos dados (para que o Projeto A nunca veja os dados do Projeto B).

Controle de Acesso: É aqui que fica o botão de "emergência" para desativar a instituição caso a mensalidade não seja paga, bloqueando imediatamente o token de acesso.

Módulo 2: Gestão de Conviventes (O Prontuário)
O módulo de entrada e acompanhamento cadastral.

O que faz: Ficha completa de quem entra no projeto. Nome, contatos de emergência, histórico médico básico e, o mais importante, o Status (Ativo, Em Fuga, Alta, Transferido).

Métricas: Calcula automaticamente o tempo de permanência com base na data de entrada.

Módulo 3: Logística e Acomodação (Gestão de Espaço)
Módulo focado no espaço físico da instituição.

O que faz: Cadastro de quartos, alas ou casas do projeto e suas respectivas capacidades (ex: Quarto 1 - 4 camas).

Alocação: Uma interface visual onde você aloca ou transfere o convivente de um quarto para outro, mantendo o histórico de por onde ele passou.

Módulo 4: Rotina Diária (O Motor Offline-First)
Este é o módulo que os funcionários mais vão usar no dia a dia, e o que mais precisa ser rápido e leve.

O que faz: Controle de presença no refeitório, frequência em atividades terapêuticas ou laborais e anotações rápidas de comportamento.

Engenharia: É aqui que a mágica do PWA (Progressive Web App) acontece. O funcionário faz a chamada da janta pelo celular, o React salva no localStorage e manda para o Python em segundo plano.

Módulo 5: Faturamento (O seu Motor Financeiro)
A parte que garante a saúde financeira do seu SaaS através da gestão das assinaturas.

O que faz: Integração com os gateways de pagamento, controle de vencimentos e geração de faturas mensais para as instituições.

Lógica de Cobrança: Ao estruturarmos o motor financeiro, a lógica precisará ser exata. O sistema deve considerar como "projeção" estritamente os itens com datas futuras em relação ao dia de hoje. Durante a conciliação de assinaturas ou dívidas ativas da instituição, essas parcelas projetadas devem ser ignoradas. Isso garante que, se você ou o cliente tentarem puxar um relatório de conciliação hoje, o sistema não some parcelas que ainda não fazem parte do ciclo atual, o que inflaria erroneamente os totais em aberto.
